--Synchronized vs locks:-
----locks have tryLock() and fairness-(long waited thread gets the lock)
----locks interface is more flexible as lock can be acquired in one method and can be released in other, which is not possible with synchronized.(reentrant lock)

--ConcurrentHashMap is a thread safe java collection which provides high concurrency by providing segment/bucket level locking instead of locking the entire map.
----It allows multiple threads to read without blocking and write concurrently to different segments/buckets.
----Retrievals (get, containsKey) never lock.
----Updates (put, remove) lock only a small portion of the table.
----A read reflects the most recently completed update at the time the read begins, Formally, an update to a key happens-before a successful retrieval of that key.
----Bin based architecture and a bin can be list,tree or empty. updates will lock the only bin being modified.Other bins remain accessible to other threads, multiple threads can update different bins simultaneously.
----When multiple keys map to the same index,i.e. collision happens then the bin can be converted to balanced Tree.
----Fail safe and Do not throw ConcurrentModificationException.
----null is used internally as a reliable atomic indicator of “no result”.

--The CopyOnWriteArrayList in Java is a thread-safe implementation of the List interface found in the java.util.concurrent package. 
----It is designed for environments where read operations vastly outnumber write operations.
----It achieves thread-safety by creating a fresh copy of the underlying array for every modification operation (add, set, remove, etc.). 
----This ensures that concurrent read operations on the original array remain unaffected. 
---- Both Concurrent HashMap and CopyOnWriteArrayList uses volatile variables and concurrent HashMap uses CAS (compare and swap) methods.

--CAS vs. synchronized:-
----When inserting the first element into an empty bin, ConcurrentHashMap uses CAS via Unsafe to place the node directly.
----CAS allows threads to attempt modifications without locking the entire table or bin first.
----Empty Bins: If a bin is empty, a CAS operation is attempted to insert a new node, avoiding locks.
----Occupied Bins: If the bin is not empty, synchronized is used on the first node of that bin to handle subsequent operations.
----Performance: CAS is generally faster as it avoids the overhead of traditional locking, which is why it is preferred for initial, uncontended insertions. 
----CAS allows for higher throughput by enabling concurrent writes to different, non-empty buckets while using synchronized only when necessary for collision handling.
